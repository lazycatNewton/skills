# A股情绪复盘

## 核心定位

将本技能视为 A 股情绪复盘工作流，而不是交易信号生成器。输出应是可解释的每日复盘和精简的次日观察池。不要提供直接买入或卖出指令。

## 数据流程

1. 明确交易日期，格式为 `YYYYMMDD`。如果用户使用“今天”等相对日期，必须同时写出具体日期。
2. 默认使用者所在 Agent 已安装所需 MCP 服务。Agent 负责调用 MCP 获取数据，Python 脚本只接收 MCP 返回的原始 payload 并执行整理逻辑。
3. 以 `mcp_router.get_limit_pool` 且 `pool="both"` 作为涨停/跌停数据的主数据源。
4. 获取涨跌停数据后，不要在对话中展开完整 payload；优先将 MCP 原始 payload 直接传给 `scripts/bridge_limit_pool_payload.py` 保存为 raw JSON，并生成整理后的 JSON。
5. 完成 raw payload 保存后，必须先整理行业/板块视图和连板梯队视图，再进入个股筛选。
6. 生成第三节 `连板梯队` 时，优先使用 `scripts/stock_emotion_report.py --organized output/data/stock-emotion-capture/{YYYYMMDD}-organized.json --section ladder` 生成矩阵表，避免手工改写格式。
7. 如果 `get_limit_pool` 返回空、失败、日期非交易日，或收盘前数据明显不完整，不要硬凑复盘；必须说明数据状态，并按降级规则输出。
8. 需要指数背景时，使用 `mcp_router.get_market_index`。
9. 只有在需要历史确认时，才使用 `mcp_router.get_historical_data`。
10. 将涨跌停池中的 `industry` 作为第一层行业/题材字段。如果没有更强的概念数据源，必须明确说明。
11. 默认不纳入 ST 股和新股整理；只有使用者明确声明“包括ST和新股”时，才将 `is_st=true` 或 `is_new_stock=true` 的股票纳入行业/板块视图、连板梯队视图和后续筛选。
12. 不要编造缺失字段、题材、板块地位或上涨原因。不确定判断必须标注为推断。

字段定义和派生指标见 `references/mcp_router_contract.md`。

## 复盘流程

按以下顺序分析：

1. 市场情绪：涨停数、跌停数、连板高度、连板数量、炸板风险、封板质量、指数背景。
2. 板块情绪：行业/题材集中度、连板股分布、龙头/中军/跟风结构、持续性、分歧或退潮信号。
3. 个股强度：封板时间、回封质量、封单强度、换手率、成交额、连板高度、ST/次新风险。
4. 风险过滤：剔除或降级 ST、异常次新、尾盘弱封、反复炸板、封单明显不足、孤立无板块支撑的个股。
5. 次日预案：输出次日观察条件和失效条件，而不是买入建议。

评分口径和报告格式见 `references/review_framework.md`。

## 必需输出

除非用户另有要求，否则按以下顺序输出：

1. 市场情绪结论
2. 主线板块复盘
3. 连板梯队
4. 涨停结构分析
5. 风险信号
6. 次日观察池
7. 个股入选理由与失效条件

第三节 `连板梯队` 必须使用矩阵表：左侧第一列为 `n板`，上侧表头为 `industry` 行业口径，单元格填写对应股票；默认只展示 2 板及以上连板股。

观察池要保持精简。默认保留 5-15 只候选，并按 `核心观察`、`重点观察`、`风险观察`、`剔除` 分组。弱市、退潮、数据不足或候选质量不达标时，允许少于 5 只或空观察池，不能为了凑数量降低标准。

## 落盘规则

最终复盘结果必须保存为 Markdown 文件。保存路径使用项目相对路径：

```text
output/reports/{YYYYMMDD}.md
```

`{YYYYMMDD}` 必须与本次复盘的交易日期一致，例如 `output/reports/20260702.md`。不要使用 `YYMMDD` 或其他日期格式。

完成复盘时，即使已经在对话中展示报告，也必须创建或更新该 Markdown 文件。最终回复必须说明实际保存路径。

## 完整性规则

- 避免未来函数：不得用次日结果解释当日入选逻辑。
- 保持原始数据和加工结论的边界：实际 MCP 字段、派生指标、主观推断必须分开表达。
- 明确说明数据缺口，尤其是板块概念、封单口径、实时/收盘口径、节假日或未收盘数据。
- 没有新闻、公告或可靠概念数据时，不得写“受某消息刺激”“资金认可某概念”等因果句，只能写基于涨跌停池和行业口径的观察。
- 优先使用简单、可审计的规则，再考虑复杂评分。
- 所有结论都必须能追溯到源数据字段，或追溯到明确命名的派生计算。
